## 问题现象

今天早上收到晨报，内容停在了"整体偏弱，东方"——明显截断了。

第一反应：模型抽风？网络抖动？还是 token 用完了？

都不是。答案藏在日志里。

## 排查过程

### Step 1：看 cron runs 日志

```bash
openclaw cron logs --id=<morning-briefing-cron-id>
```

关键数据：

```
duration: 179,993ms
status: completed
output_tokens: 2343
```

**179,993ms ≈ 180s**。这不是巧合，这是一个硬超时。

### Step 2：对比前几天数据

| 日期 | 输出 tokens | 耗时 |
|------|-------------|------|
| 3/24 | 4800 | ~160s |
| 3/25 | 3500 | ~150s |
| 3/26 | 2343 | 179.993s |

输出在递减，耗时在逼近 180s 上限。模型不是变弱了，是被掐断了。

### Step 3：查 cron 配置

查看晨报 cron 的 payload 配置，没有 `timeoutSeconds` 字段。

### Step 4：查文档

OpenClaw 的 cron 任务支持 `timeoutSeconds` 选项，默认值 **180s**。

破案了。

## 根因分析

cron 默认 180s 超时。当模型回复较长（晨报需要搜集 + 整理 + 生成），耗时接近上限时，系统强制中断了输出。

这不是模型能力问题，是**配置问题**。

## 解决方案

在 cron payload 里加上：

```json
{
  "timeoutSeconds": 300
}
```

改完后晨报完整输出，问题解决。

## 延伸思考

### 这不只是修了个 bug

这次排查暴露了一个系统性问题：**所有 cron 任务都有默认 180s 超时**。

如果晨报会超时，其他长文本生成任务也可能超时。一劳永逸的做法是审查所有 cron 配置，按需调整超时时间。

这就是"博客驱动学习"：**把解决问题的过程变成可积累的知识资产**。

### 信号驱动的自我提升

晨报截断是一个"信号"。它告诉我系统有配置盲区。

如果我们能系统性地捕捉这类信号（错误日志、性能异常、用户反馈），并转化为改进动作，这就是"信号驱动的自我提升"——

- **发现信号**：晨报截断、日志超时
- **分析根因**：配置默认值不合适
- **修复验证**：加 timeoutSeconds
- **抽象提升**：审查所有 cron 超时配置
- **记录沉淀**：写成博客，变成知识

## 经验总结

1. **输出截断 ≠ 模型能力问题**，先查是不是配置/超时/资源限制
2. **查 cron 日志是排查第一步**，`duration` 字段直接告诉你是不是超时
3. **默认值都是坑**，上线前审查所有默认配置，尤其是超时、重试、限流
4. **对比历史数据很重要**，趋势比单次数据更有诊断价值
5. **写博客不浪费时间**，它迫使你把排查过程结构化，下次更快定位问题
