Lessons — 教训汇总

从实际事件中提炼的教训，按类别组织
来源：improvement-log.md 验证通过后的固化

配置类

  1. 默认值不适合生产（3/27）：cron 180s 超时导致晨报截断；所有默认配置需人工审查
  2. feature 配置 ≠ feature 可用（3/27）：配了搜索但没装插件，hexo generate 才能验证

执行类

  1. 不能等指令，要主动迭代自己（3/27）：有搜索能力≠用搜索能力，每次心跳应主动探索
  2. 变更操作必须走Superpowers流程（3/25）：Brainstorm→Plan→Execute→Verify，一个阶段都不能跳
  3. 图快反而慢（3/25）：急着给结果→漏步骤→返工→更慢。慢就是快
  4. 诊断先验证再结论（3/25）：有搜索能力和命令行，先跑验证、先搜社区经验，不要凭推测编故事
  5. 有搜索能力就要用（3/25）：常见问题网上都有方案，不要闭门造车
  6. 派审计先说明网络拓扑（3/23）：否则误判内网为公网
  7. 读报告再转述（3/23）：不能当传声筒，要核实

工具类

  1. 密钥不进聊天（3/21）：铁则
  2. PROXY_HOST非HTTP_PROXY（3/21）：MoviePilot专属配置
  3. Docker compose重建=镜像版本（3/21）：每次重建需重新升级
  4. Hysteria2证书三要素（3/22）：IP SAN + server_name + insecure

记忆类

  1. 派任务前先搜索已有资源（3/27）：不要凭记忆造目录，find/glob 先行
  2. heartbeat-state.md 必须初始化（3/28）：skill 安装后要跑 setup，模板≠实例
  3. 信号检测必须闭环（3/28）：检测到信号→记录→确认写入成功，不能只检查不记录
  4. 写了不读 = 没写（3/29）：方案写了功能点但派工没核对，日记写了教训但下次没看。记忆文件必须作为每次派工/验收的 checklist 强制引用
  5. 派工 checklist 必须逐项勾对（3/29）：方案里每个 P0/P1 功能点都要在派工时明确列出，不能凭印象跳过

子Agent类

  1. 子agent写文件必须验证（3/29）：hunter模型会生成回复但不调write工具，spawn后必须ls/cat确认文件落盘，连续失败就自己干
  2. 前端工作必须用frontend-design skill（3/29）：避免AI化设计（蓝色渐变毛玻璃），用skills/frontend-design-3保证设计质量

自进化类

  1. 自进化 = 主动发现问题（3/29）：不是等用户喂信号。心跳必须扫描系统健康、对话纠错、项目一致性，不能只读状态文件说”无信号”
  2. 写了规则不执行 = 没写（3/29）：HEARTBEAT.md 写了主动扫描但实际只做被动检查，等于没写

API/工具类

  1. 先用最小参数测试API（4/2）：CogVideoX 不支持自定义 fps/duration，传了就400。应先不传任何可选参数确认默认行为，再逐步加参数
  2. 容器内绑定127.0.0.1=外部不可达（4/2）：mimo-tts-proxy 监听 loopback，端口映射失效。部署容器时确认监听地址是 0.0.0.0
  3. Python 3.11 f-string 不支持反斜杠（4/2）：f"{d[\"key\"]}" 报 SyntaxError，需提前提取变量
  4. 需求先复述确认（4/2）：sigh说”猫狗嘲笑主人”，我脑补了主人角色，实际只有猫狗。拿到需求先复述一遍再动手
  5. 静态图≠短剧（4/2）：Ken Burns 推拉没有表情/动作/口型，无法形成生产力。AI短剧必须上视频生成
